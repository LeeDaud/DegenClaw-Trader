# 预测机制重设计

## 0. 核心判断

> **涨跌预测频繁翻转的原因是：阈值太宽、窗口太短、用价格预测价格。**
>
> 套利逻辑是对的——Agent 短时交易表现改善 → 排名上升 → 市场未完全定价 → 提前发现买入 → 市场跟进。信息差套利的核心是比别人更快发现真实的趋势转变，而不是对每一分钟的噪声做出反应。

---

## 1. 信息差套利的完整链条

```
Agent 发生真实改善（交易策略有效、PnL 持续上升）
  → DegenClaw 排行榜排名上升
  → 市场意识到该 Agent 被 AI Council 选中的概率提升
  → Agent Token 价格开始反映这个预期
  → 如果 Agent 最终被 Council 选中，AI Pot copy-trade → 更多买盘
  → 如果 Agent 持续表现好，Pot 盈利 → buyback → 更多买盘

我们的位置：
  采集（60s）→ 检测到改善趋势（15-30min 确认）→ 提前于市场大部分参与者
```

**关键：我们不需要预测终极结果（是否被选中、是否 buyback），只需要比市场更快地发现改善趋势。** 趋势持续的时间窗口是小时级到天级，够了。

## 2. Root Cause 修正分析

### 2.1 PnL 信号：方向对，窗口和阈值错

```
当前：
  60s 采一次 snapshot → latest.pnl_24h - prev.pnl_24h = "变化"
  ≥ 8% → surge / ≤ -8% → dump

问题：
  pnl_24h 本身是 24 小时滚动值，60 秒前后差异极小
  "pnl_24h ≥ 8%" 是一个静态状态，不是变化
  真正的改善信号是 pnl_7d 的变化趋势

实测场景：
  Agent A 的 24h PnL 从 +7.9% → +8.1%（60 秒内因 1 笔浮盈变化）
  → 触发 "收益暴涨" 
  下 60 秒回到 7.9% → "信号消失"
  → 翻转
```

### 2.2 排名信号：2 快照比较导致对振荡敏感

```
当前：
  snapshots[0] vs snapshots[1]（间隔 60 秒）
  差 ≥ 5 位 → rank_surge（注意：5 位在同级竞争中不罕见）

问题：
  Agent 在 #15-#17 之间波动是正常的（多个 Agent 排名接近，交易更新顺序影响）
  单次 5 位"跳变"可能只是数据刷新顺序导致的假象，不是真实趋势

实测场景：
  T+0: rank=15, T+1: rank=20（差 5 位）→ "排名暴跌" 
  T+2: rank=16（差 4 位）→ "排名恢复"
  → 翻转
```

### 2.3 价格信号：用价格预测价格（循环论证）

```
当前：
  price_change_1h ≥ 10% → "价格暴涨" 看涨
  price_change_1h ≤ -8% → "价格暴跌" 看跌

问题：
  这是追涨杀跌，不是预测
  Agent Token 是低流动性 meme token，1h ±20% 是常态
  "刚才涨了所以还要涨" 没有因果逻辑支撑

注意：价格数据仍然重要 —— 价格反映了市场预期是否已提前兑现
  比如：Agent 排名改善但价格没动 → 市场还没发现 → 套利空间大
  比如：Agent 排名改善但价格已经涨了 80% → 预期已兑现 → 不应追高
  价格是 "参考系"，不是 "预测量"
```

### 2.4 综合看涨/看跌：大杂烩无方向记忆

```
当前：
  每次从零算 combined_up/combined_down
  13 个因子权重大杂烩，总分数在阈值 35 附近振荡

问题：
  没有人能解释"分数从 36 掉到 34"具体是哪个因子变了
  振荡时每次触发方向都不同
  综合分应该用于评分引擎（趋势性），不该用于信号引擎（事件性）
```

## 3. 修正后的信号体系

### 3.1 设计原则

1. **保留信息差套利的信号类型**（PnL surge、rank surge 是有意义的），但修正检测方式
2. **拉长检测窗口**：用趋势检测替代瞬时比较（3-5 采样点滑动窗口，覆盖 3-5 分钟）
3. **提高有效变化阈值**：区分"正常波动"和"真实改善"
4. **移除价格方向信号**（`price_surge`/`price_dump`），改用价格做"预期兑现参考"
5. **每次只推一个方向**：同一个 Agent 同一轮只推最多一个方向的信号（优先级：暴涨 > 暴跌 > ...）

### 3.2 信号类型保留与调整

| 信号 | 保留？ | 调整内容 |
|------|--------|---------|
| `rank_surge` | ✅ | 改趋势检测：5 采样点中 4 个处在改善通道，非单次 5 位跳变 |
| `rank_dump` | ✅ | 同上，改趋势检测，提高阈值到 8 位（正常波动范围更大） |
| `surge`（PnL 爆发） | ✅ | 改检测周期：pnl_7d 的趋势变化 + pnl_24h 阈值提到 15%+ |
| `dump`（PnL 暴跌） | ✅ | 同上，阈值降到 -12%（-8% 太常见） |
| `price_surge` | ❌ 移除 | 用价格预测价格，无因果逻辑 |
| `price_dump` | ❌ 移除 | 同上 |
| `volume_spike` | 保留但不推送 | 成交量是辅助信号，不独立触发预警 |
| `wr_surge` / `wr_dump` | ✅ | 保留但窗口拉长到 5-10 个快照 |
| `combined_surge` / `combined_dump` | ✅ 保留但重构 | 改用评分引擎分数 + 滑动窗口 EMA |

### 3.3 价格信号降级为"预期兑现参考"（非推送信号）

```
Agent 排名上升且价格未动：
  → 套利空间：市场未发现 → aggressively signal
Agent 排名上升但价格已涨：
  → 预期已部分兑现 → 降低推送优先级
Agent 排名下降但价格没跌：
  → 价格滞后，可观望
Agent 排名下降且价格已跌：
  → 确认恶化 → signal

模式：price 作为校验维度，不作为独立信号
```

### 3.4 趋势检测算法

```
当前（错误）：
  change = snapshot[0] - snapshot[1]  （60 秒视角）

改进（正确）：
  readings = [snapshot[0], snapshot[1], ..., snapshot[N]]  （N ≥ 3，覆盖 3+N 分钟）
  
  # 趋势方向判断：多数采样点一致
  if 80%+ 的连续方向读数一致 → "确认趋势"
  
  # 趋势强度判断：
  中位数变化幅度 ≥ 阈值 → "有效趋势"
  
  # 例子：5 个采样点，rank 序列 [15, 16, 18, 19, 20]
  方向一致占比 = 4/4 = 100%（全部向下）
  中位数变化 = 4 位（从 #15 到 #19 的中位数）
  → 确认 rank_dump

  对比当前做法：
  [15, 20] → change = 5 → dump
  [20, 16] → change = -4 → 无信号（但不是因为恢复了，是采样抖动）
```

### 3.5 单 Agent 单轮单方向规则

```
一轮检测中，同一 Agent 的信号：
  1. 计算 bullish 信号的总分
  2. 计算 bearish 信号的总分
  3. 如果两边都有：
     - 分开评估确认度（方向确认计数）
     - 只推送确认度更高的一方
     - 如果确认度相同 → 都不推送

目的：避免同一 Agent 同时收到"综合看涨"和"排名暴跌"
```

## 4. 实现计划

### Phase A：趋势检测算法（核心改造）

- [ ] 修改 `_analyze()` 的排名检测：用 N 个快照的趋势替代单次差值
- [ ] 修改 PnL 检测：pnl_7d 趋势 + pnl_24h 阈值提升
- [ ] 修改 win_rate 检测：窗口从 3 个快照扩展到 5-10 个
- [ ] 实现"多数一致"判定逻辑（滑动窗口趋势确认）

### Phase B：移除和降级信号类型

- [ ] 移除 `price_surge` / `price_dump` 信号生成
- [ ] 价格数据保留在 Alert 的 snapshot_data 中做"预期兑现参考"
- [ ] `volume_spike` 降级为辅助数据，不推送
- [ ] 弃用 `SIGNAL_THRESHOLDS` 中的价格相关阈值

### Phase C：单 Agent 单轮单方向

- [ ] `_build_aggregated_alert()` 中增加方向冲突裁决
- [ ] 两边信号共存时只推确认度更高的一方
- [ ] 确认度相同时两方都不推

### Phase D：阈值调整

- [ ] `pnl_surge_min_24h`: 8.0 → 15.0
- [ ] `pnl_dump_max_24h`: -8.0 → -12.0
- [ ] `rank_surge_min`: -5 → 趋势检测 80%+ 一致
- [ ] `rank_dump_min`: 5 → 趋势检测 80%+ 一致

### Phase E：验证

- [ ] 部署观察 24h，统计每类信号的推送频次
- [ ] 确认无同 Agent 同轮双向推送
- [ ] 确认翻转率降到 0
- [ ] 调整趋势检测窗口大小和一致比例参数
